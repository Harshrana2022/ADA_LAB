## 3. OBJECTIVE / SCOPE

The objective of The Book Nook project is to develop a comprehensive, scalable e-commerce platform that revolutionizes the online book retail experience by leveraging modern software architecture paradigms and user-centric design principles. The project aims to bridge the gap between established e-commerce best practices and contemporary distributed systems architecture, creating a robust, maintainable platform that can serve as both a functional application and a reference implementation of microservices design patterns.

The primary goal is to create a complete end-to-end solution that demonstrates professional software engineering practices including proper separation of concerns, secure authentication mechanisms, and data integrity management. Rather than serving as a toy application, The Book Nook is designed to be production-ready, with extensibility built into its foundation to accommodate future features such as advanced payment integrations, recommendation engines, and expanded analytics capabilities.

### Project Scope

The project encompasses the following key dimensions:

**Backend Infrastructure Development**: Design and develop a microservices-based backend architecture consisting of six independent, loosely-coupled services that communicate through well-defined RESTful APIs. Each microservice is responsible for a specific business domain, allowing for independent development, testing, deployment, and scaling. The Service Registry provides dynamic service discovery, eliminating the need for hardcoded service endpoints and enabling seamless horizontal scaling.

**Frontend Application Development**: Implement a modern, responsive React-based user interface built with TypeScript support, ensuring type safety and improved developer experience. The frontend is structured to provide two distinct interfaces—one for end-users engaging in book shopping activities and another for administrators managing inventory, users, and business metrics. Both interfaces prioritize user experience through responsive design that adapts seamlessly across desktop, tablet, and mobile devices while maintaining accessibility compliance with WCAG 2.1 standards.

**Security Implementation**: Establish a secure, stateless authentication system using JWT (JSON Web Tokens) with BCrypt password hashing, eliminating server-side session storage requirements and enabling scalable distributed deployments. Role-based access control distinguishes between USER and ADMIN roles at both API and frontend levels, ensuring that sensitive operations are properly protected and user data remains secure throughout the system.

**Book Catalog and Inventory Management**: Develop a comprehensive book catalog system supporting detailed product information including title, author, ISBN, category, pricing, stock quantities, and discount management. The system enables administrators to perform complete CRUD operations on the inventory, ensuring that the catalog remains current and accurate while providing customers with comprehensive search and filtering capabilities to discover desired books efficiently.

**Shopping and Payment Processing**: Implement a secure shopping cart system allowing customers to accumulate multiple books for purchase, with real-time price calculations considering applicable discounts. The order management system handles complex order structures supporting multiple items per order while maintaining referential integrity. The payment service provides transaction processing, status tracking, and audit logging for compliance and troubleshooting purposes.

**Admin Analytics Dashboard**: Create a business intelligence dashboard that provides administrators with comprehensive insights into business performance through real-time metrics, revenue tracking, sales analysis, customer management, and payment monitoring. The dashboard features animated counters and visual representations of key performance indicators enabling data-driven decision making.

**Containerization and Deployment**: Package each microservice and the frontend application as Docker containers with optimized configurations, enabling consistent deployment across development, staging, and production environments. This approach eliminates "works on my machine" problems and facilitates rapid deployment pipelines and horizontal scaling.

**Service Discovery and API Gateway Patterns**: Implement the Service Registry pattern using Netflix Eureka to provide dynamic service discovery, and the API Gateway pattern using Spring Cloud Gateway to serve as a single entry point for all client requests. These patterns abstract away the complexity of managing multiple microservice endpoints and enable advanced features like cross-origin resource sharing configuration and request routing.

The scope deliberately focuses on delivering a complete, functional e-commerce platform rather than addressing every possible feature. However, the architecture is designed with extensibility as a core principle, allowing for future enhancements such as advanced search capabilities, machine learning-based recommendations, multiple payment providers, and expanded reporting features without requiring architectural redesign.

## 7. EXECUTED WORK

The Book Nook project has been successfully developed through systematic implementation following a structured development lifecycle emphasizing clean architecture, comprehensive testing, and production-readiness. The following section provides detailed documentation of completed work, achieved results, and delivered components.

### 7.1 Backend Infrastructure Development

**Service Registry Implementation**: The Service Registry component serves as the central nervous system of The Book Nook's microservices architecture. Implemented using Spring Cloud Netflix Eureka, this component maintains a continuously updated registry of all available microservices, their network locations, and health status. The implementation includes automatic service registration when microservices start up, automatic deregistration when services stop, and periodic health checks every 30 seconds to verify service availability. The registry uses client-side load balancing, where the API Gateway queries Eureka to discover service instances and distributes requests across multiple instances when available.

The Eureka server dashboard provides operators with real-time visibility into service health, displaying which services are registered, how many instances exist for each service, and the current status of each instance. This visibility enables rapid identification of service failures and facilitates understanding of system state during operational issues. The implementation supports service filtering by name and status, enabling targeted investigation of problems. The registry includes configurable timeout thresholds, allowing adjustment of how quickly failed services are removed from the registry based on operational requirements.

**API Gateway Development**: The API Gateway, implemented using Spring Cloud Gateway with a reactive WebFlux backend, serves as the single entry point for all client requests from the React frontend. This strategic architectural decision centralizes request handling, enabling uniform application of cross-cutting concerns and eliminating the need for clients to discover individual service locations. The gateway implements sophisticated routing rules that examine incoming requests and forward them to appropriate backend services based on URL path patterns.

The CORS (Cross-Origin Resource Sharing) implementation explicitly allows requests from the React frontend running at http://localhost:3000, enabling frontend-to-backend communication across origins. The configuration specifies allowed HTTP methods (GET, POST, PUT, DELETE, PATCH, OPTIONS), permitted headers (all headers including Content-Type and Authorization), and support for credentials including cookies. This configuration balances security (restricting to known origins) with functionality (allowing necessary methods and headers).

The gateway's reactive architecture processes requests asynchronously without blocking, enabling efficient handling of high request volumes and multiple concurrent connections. The WebFlux implementation uses Project Reactor underneath, providing powerful stream processing capabilities and backpressure handling. The gateway dynamically discovers backend services through Eureka integration, automatically adapting to services coming online or going offline without requiring gateway configuration changes or restarts.

### 7.2 User Service Implementation

**User Authentication System**: The User Service implements a comprehensive authentication and authorization system enabling secure user registration, login, and session management. The registration endpoint accepts user details including name, email, password, and role assignment, validating inputs against constraints ensuring data quality. Email uniqueness is enforced at the database level through unique constraints, preventing account duplication. Passwords undergo multiple transformation steps: first, Spring Security validators ensure passwords meet minimum length requirements (6 characters minimum); second, BCryptPasswordEncoder hashes passwords with configurable strength (default 12 rounds), generating a unique salt for each password; third, only the hashed password is persisted to the database.

The login endpoint validates supplied credentials by retrieving the user record by email and comparing the supplied password against the stored hash using BCrypt's timing-safe comparison function, preventing timing-based attacks. Upon successful authentication, the JwtUtil component generates a JWT token containing the user's email and role, signed with the application's secret key. This token is returned to the client, where it is stored and included in subsequent API requests, enabling stateless authentication without server-side session storage.

**User Management Capabilities**: Beyond authentication, the User Service provides complete user management functionality. The list users endpoint enables administrators to retrieve all registered users with their details, supporting user discovery and management. The get user endpoint retrieves a specific user's profile including name, email, and role, enabling users to view their own profiles and administrators to inspect user details. The update user endpoint allows administrators to modify user information including name, email, and role assignment, with proper authorization checks ensuring that regular users cannot modify other users' accounts. The implementation includes special handling for password updates, only overwriting passwords when explicitly provided, allowing updates to other fields without disturbing existing credentials. The delete user endpoint enables account removal, with proper cascade handling of related data.

**Role-Based Access Control**: The User Service implements two distinct roles—USER for regular customers and ADMIN for administrative staff. The role is persisted in the user record and included in JWT tokens, enabling both backend services and frontend components to make authorization decisions based on role. The implementation follows the principle of least privilege, restricting sensitive operations to the ADMIN role. User endpoints include authorization annotations preventing non-admin users from accessing sensitive operations, while allowing users to access their own profile information.

### 7.3 Order Service Implementation

**Complex Order Structure**: The Order Service manages the complete order lifecycle, handling orders containing multiple books with quantity tracking and price management. The underlying data model uses two related tables: the Orders table storing order-level information including user ID (linking to the customer), payment ID (linking to the associated payment), and total price (computed from order items); the OrderItems table storing individual items within each order, including book ID, quantity, and the unit price at the time of order (enabling historical price tracking even if catalog prices change).

The one-to-many relationship between Orders and OrderItems, with cascade delete operations, ensures that deleting an order automatically removes all associated order items, maintaining referential integrity without orphaned records. The implementation includes proper transaction handling, ensuring that either an entire order with all its items is created, or nothing is created, preventing partial order creation due to transient failures.

**Order Lifecycle Management**: The order creation endpoint accepts a complete order request including user ID, list of order items with book IDs and quantities, and enables comprehensive validation before order creation. The implementation calculates the total price from item-level prices and quantities, enabling verification that the client-side calculation matches server-side calculation. Upon successful creation, the order is persisted with an initial status and a timestamp indicating creation time.

The order retrieval endpoints enable users to view their order history with pagination support, allowing efficient navigation through orders even for customers with extensive purchase history. Each order includes the associated items with details enabling customers to understand what they purchased and for what price. The get order by ID endpoint returns complete order details suitable for display on an order confirmation or tracking page. The delete order endpoint enables order cancellation with business logic determining whether cancellation is permitted (e.g., preventing cancellation of orders already shipped or paid).

### 7.4 Payment Service Implementation

**Transaction Processing and Tracking**: The Payment Service manages payment processing with comprehensive tracking of payment status through multiple states. Each payment record includes the associated order ID, payment amount, status (PENDING, COMPLETED, or FAILED), transaction ID for external payment provider integration, and timestamps indicating creation and last modification time. The automatic timestamp tracking enables understanding of when payments were initiated and when status changes occurred, crucial for debugging and compliance.

The payment creation endpoint initializes payment records with PENDING status, enabling the order confirmation flow to progress even before actual payment processing completes. The status query endpoints enable customers and administrators to check payment status at any time, with filtering by multiple criteria (payment ID, order ID, status, transaction ID) enabling comprehensive payment investigation. The update endpoint enables status transitions, typically changing PENDING payments to COMPLETED when the payment provider confirms successful processing, or to FAILED when payment declined or failed.

**Audit and Compliance**: The implementation includes comprehensive audit capabilities through automatic timestamp tracking, enabling understanding of payment history and duration in each state. The transaction ID field enables correlation between The Book Nook's payment records and external payment provider systems, crucial for debugging failed payments and handling disputes. The immutable creation timestamp, combined with mutable updated timestamp, creates an audit trail showing when status changes occurred. This audit trail proves invaluable during payment investigations, chargebacks, or regulatory audits.

### 7.5 Frontend Application Development

**Dual-Interface Architecture**: The frontend implements two entirely separate user interfaces serving different audiences with different requirements. The customer-facing shopping portal includes the home page displaying book catalog with search and filtering, login and registration pages for authentication, shopping cart interface enabling product selection and quantity management, checkout page for order review and confirmation, order history page showing past purchases, and order summary page showing detailed order information.

The admin portal includes a comprehensive dashboard displaying key business metrics including total revenue, transaction count, best-selling books, and customer acquisition metrics; user management interface showing registered users with administrative controls; book inventory management system enabling complete CRUD operations on the catalog; and payment analytics showing transaction history, status distribution, and revenue trends. The separation of these interfaces prevents regular users from accidentally accessing administrative features while allowing administrators convenient access to management tools.

**Component Architecture and Reusability**: The frontend uses React's component-based architecture to achieve high reusability and maintainability. Radix UI provides foundational building blocks—accessible, unstyled components implementing WAI-ARIA best practices. Shadcn/ui builds on Radix UI to provide pre-styled, composable components. The development team created custom components wrapping these primitives to provide application-specific functionality. The book card component, for example, displays book information with images, pricing, and add-to-cart functionality, reused throughout the application whenever books are displayed.

The shopping cart component manages cart state, enabling users to add items, modify quantities, remove items, and proceed to checkout. The product filters component displays available filter options (by category, price range, availability) with interactive controls. Protected route components prevent unauthorized access to sensitive areas like order history and checkout pages. The implementation uses React Router for navigation, providing efficient page transitions and browser history support.

**State Management**: The frontend employs React Context API for global state management, centralizing management of authentication state (current user, JWT token), shopping cart state (items, quantities, total price), and UI state (active filters, sort preferences). Custom hooks encapsulate complex state logic and enable reuse across components. The useAuth hook manages authentication, including login, registration, logout, and token persistence. The useCart hook manages shopping cart operations including adding items, updating quantities, and calculating totals. The useFilter hook manages filter state and provides functions for filter updates.

### 7.6 Database Schema and Deployment

**Schema Implementation**: The MySQL database implements a carefully designed schema following normalization principles to ensure data integrity and query efficiency. The Users table stores user account information with proper constraints preventing invalid data. The Orders table maintains order records with foreign keys to Users and Payments tables. The OrderItems junction table enables one-to-many relationships between orders and items, with foreign keys back to Orders and references to Books. The Payments table tracks payment records with unique constraints preventing duplicate payments and enum types enforcing valid status values.

Indexes on frequently queried columns (user_id, order_id, payment_id) enable efficient query execution even with large datasets. Cascade delete operations ensure that deleting an order automatically removes associated order items, preventing orphaned records that would waste storage and complicate queries. The implementation includes check constraints ensuring that monetary amounts are positive values and that only valid enum values appear in status columns.

**Docker Containerization**: Each component of the application is packaged as a Docker container with optimized configurations. The backend services use multi-stage builds where dependencies are downloaded in one stage and the compiled application is copied into a minimal final image, reducing image sizes from potentially hundreds of megabytes to tens of megabytes. The Dockerfile for each service includes health check configurations enabling Docker to periodically execute a command (typically an HTTP health check endpoint) to verify service availability. Services failing health checks can be automatically restarted or removed from load balancers.

The frontend Dockerfile uses a multi-stage build, compiling the React application in one stage and copying the compiled static files to an Nginx server in the final stage. This approach enables serving the React application with proper HTTP caching headers and gzip compression, improving load times for end users. The Nginx configuration includes proper MIME type configuration for various file types and enables gzip compression for text resources.

**Production Readiness**: All containers include proper logging configuration, writing application logs to stdout/stderr rather than files, enabling Docker's logging infrastructure to capture and archive logs. Environment variables enable configuration without rebuilding containers, allowing the same image to be deployed across development, staging, and production environments with appropriate configuration. Security scanning tools can analyze images for known vulnerabilities before deployment, and secrets management systems can securely inject sensitive values like database passwords and JWT signing keys at runtime.

## 11. SECURITY FEATURES

The security implementation of The Book Nook encompasses multiple layers and approaches, providing defense-in-depth protection against various attack vectors and ensuring the confidentiality, integrity, and availability of user data and business-critical systems.

**JWT-Based Authentication**: The authentication system uses JSON Web Tokens (JWT), which are cryptographically signed tokens that can be verified without requiring server-side session storage. A JWT consists of three parts separated by dots: a header specifying the token type and hashing algorithm; a payload containing claims (assertions about the user, such as email and role); and a signature proving that the token has not been modified. Upon successful login, the server generates a JWT containing the user's email and role, signs it with a secret key known only to the server, and returns it to the client.

The client includes this token in subsequent API requests through the Authorization header using the Bearer scheme. Each request's token is verified by the server, ensuring it has not been tampered with and that it was generated by a legitimate authority. The stateless nature of JWT authentication enables scaling without requiring shared session storage between servers—each server can independently verify tokens without consulting a central session store. Tokens include an expiration time, preventing indefinite use of tokens from compromised devices.

**Password Security and Encryption**: The password handling implementation prioritizes security at every step. Upon registration or password change, the supplied password undergoes BCrypt hashing using a configurable cost factor (12 rounds by default), which exponentially increases the computational time required to compute the hash for a given password. Each password hash includes a unique salt generated by BCrypt, preventing the use of precomputed hash tables (rainbow tables) in brute-force attacks. The application stores only the hash—never the plaintext password—so even complete database compromise does not reveal user passwords.

During login, the supplied password is never stored, logged, or transmitted except over HTTPS. The login endpoint accepts the password, compares it against the stored hash using BCrypt's comparison function, which is implemented to prevent timing attacks where a password differing in the first character would fail faster than a password differing in the last character, potentially revealing information about the correct password. The entire login operation completes in constant time regardless of where password mismatch occurs.

**Role-Based Access Control**: The system implements role-based access control (RBAC) where users are assigned roles (USER or ADMIN) that determine what operations they can perform. This principle of least privilege ensures that users have only the minimum permissions necessary for their role. USER role members can view their own profile and purchase history but cannot access user management functions. ADMIN role members can access management interfaces for users, books, payments, and analytics. The role is included in the JWT token, enabling rapid authorization decisions without consulting the database for every request.

**CORS Security Configuration**: Cross-Origin Resource Sharing (CORS) is configured to explicitly allow requests from the React frontend domain (http://localhost:3000) while denying requests from unknown domains. This prevents malicious websites from tricking user browsers into making requests to The Book Nook APIs. The CORS configuration explicitly specifies allowed methods (GET, POST, PUT, DELETE, PATCH, OPTIONS) preventing unexpected method types. While this is a development-time configuration, production deployment would replace the hardcoded localhost origin with the production domain name.

**Data Validation and Input Sanitization**: Every input to the system undergoes validation at multiple layers. The frontend validates inputs before submission, providing immediate feedback to users. The backend re-validates all inputs using Spring Validation annotations, treating frontend validation as a UX enhancement but not relying on it for security. Validation rules prevent obvious invalid values—emails must match email format patterns, passwords must meet minimum length requirements, monetary amounts must be positive, and quantities must be positive integers. This validation prevents obvious invalid data from reaching the database and prevents certain classes of injection attacks.

## 12. DESIGN PATTERNS & BEST PRACTICES

The Book Nook implements numerous software design patterns and architectural best practices that ensure code quality, maintainability, and extensibility.

**Microservices Architecture Pattern**: The microservices pattern decomposes the system into multiple independent services, each owning its business domain and data. This pattern enables different services to be developed, tested, deployed, and scaled independently, allowing teams to work in parallel without coordinating across large teams. The trade-off is increased operational complexity—distributed systems are harder to debug, require sophisticated monitoring, and introduce network latency. However, for a system like The Book Nook with distinct business domains (user management, order processing, payment handling), the benefits clearly outweigh the costs.

**Service Registry Pattern (Eureka)**: The Service Registry pattern eliminates the need for clients to know the network addresses of servers they communicate with. Instead, servers register their addresses with a central registry at startup, and clients query the registry to discover server locations. This enables dynamic deployment—services can be added or removed without changing client configuration. The Netflix Eureka implementation provides proven reliability and extensive production experience in large-scale systems.

**API Gateway Pattern**: The API Gateway pattern provides a single entry point for all client requests, rather than requiring clients to communicate directly with multiple backend services. The gateway handles cross-cutting concerns like CORS, authentication, logging, and rate limiting uniformly for all requests. The gateway also abstracts away the complexity of service discovery and multiple service endpoints—clients interact with a single gateway endpoint rather than discovering and managing multiple service addresses.

**Repository Pattern**: The repository pattern abstracts data access operations behind a consistent interface, enabling swapping of data access implementations without changing business logic. Each service uses Spring Data JPA repositories providing standard operations (save, delete, find, etc.) against the database. The repository pattern enables testing business logic without a real database by substituting a mock repository in tests.

**DTO (Data Transfer Object) Pattern**: The DTO pattern defines objects specifically for transferring data between layers, decoupling the external API from internal implementation details. The API returns DTOs containing only the data needed by clients, rather than exposing complete entity objects that might contain sensitive fields or unnecessary relationships. DTOs provide a stable API that can evolve independently from internal implementations.

**Entity Pattern with JPA**: The entity pattern defines persistent objects mapped to database tables through annotations, enabling object-oriented database access. JPA entities maintain a clear separation between the database schema and object-oriented representation, automatically handling the impedance mismatch between relational and object-oriented data models. Entities define relationships between objects, which JPA translates into foreign key constraints and join operations.

**Dependency Injection**: The application uses Spring's dependency injection container to manage object creation and lifecycle. Components declare their dependencies through annotations (e.g., @Autowired), and Spring automatically provides instances, enabling loose coupling and facilitating testing through dependency substitution.

**MVC (Model-View-Controller) in React**: The React frontend follows MVC separation, dividing components into models (state and business logic), views (rendered components), and controllers (event handlers updating state). This separation improves testability and maintainability by isolating state management from rendering logic.

**Observer Pattern**: React's hooks and Context API implement the observer pattern, where components subscribe to state changes and automatically re-render when state changes. This reactive approach eliminates boilerplate code required in more imperative UI frameworks.

**Singleton Pattern**: Spring beans are singletons by default, instantiated once and reused throughout the application. This pattern is appropriate for stateless services and repositories, reducing memory usage and improving performance.

## 14. DEPLOYMENT & CONTAINERIZATION

The deployment strategy emphasizes consistency across environments, rapid deployment cycles, and operational simplicity through containerization.

**Docker Containerization Strategy**: Each component of The Book Nook—the six backend microservices and the frontend React application—is packaged as a separate Docker container. This strategy provides several benefits: consistency (the same image runs identically across development, staging, and production); isolation (each service has its own dependencies without interference); scalability (containers can be quickly replicated); and portability (containers run on any system with Docker installed).

The containerization approach implements multi-stage builds, where intermediate build stages produce compiled artifacts, and final stages copy artifacts into minimal runtime images. For Java services, the compile stage includes the full JDK and Maven, while the runtime stage includes only the JRE and compiled JAR, reducing final image size by over 50%. For the React frontend, the build stage includes Node.js and build tools, while the runtime stage includes only Nginx and compiled static files.

**Service Configuration and Environment Variables**: Rather than hardcoding configuration values, containers accept configuration through environment variables set at runtime. Database connection strings, service port numbers, JWT secret keys, and other configuration values are injected at container startup, enabling the same image to operate across different environments without modification or rebuild.

**Health Checks and Container Orchestration**: Each container includes health check configurations enabling Docker and orchestration platforms to monitor service health. Health checks typically invoke HTTP endpoints that return success or failure, enabling automatic detection of degraded services. Orchestration platforms can use health check results to automatically restart failed containers or remove unhealthy containers from load balancers.

**Logging and Monitoring**: Containers write application logs to stdout and stderr rather than files, enabling Docker's logging infrastructure to capture, buffer, and forward logs to centralized logging systems. This approach simplifies container deployment—no log volume configuration required—and enables sophisticated log aggregation and analysis.

## 17. TECHNICAL CONSIDERATIONS

The implementation of The Book Nook involved careful consideration of multiple technical dimensions affecting system performance, reliability, and operational feasibility.

**Scalability Through Independent Service Scaling**: The microservices architecture enables horizontal scaling of individual components based on load. If the Order Service experiences high load while the User Service is underutilized, additional Order Service instances can be deployed without scaling other services. The Eureka service registry automatically incorporates new service instances into load balancing. The API Gateway distributes requests across multiple instances of each service using client-side load balancing. This targeted scaling approach optimizes resource utilization and cost compared to monolithic applications where all components must scale together.

**Stateless Service Design for Horizontal Scaling**: The services are designed to be stateless—they do not store user sessions or other mutable state locally. All state is persisted to the MySQL database. This enables any number of service instances to be created or destroyed without loss of functionality, as each instance can service any request. The JWT authentication mechanism reinforces statelessness—the client includes authentication information with every request, eliminating the need for server-side session storage.

**Reliability Through Service Isolation and Resilience**: The microservices design provides fault isolation—failures in one service do not necessarily cascade to others. If the Payment Service experiences an outage, users can still browse and add items to shopping carts, with order placement deferred until the Payment Service recovers. The Eureka registry detects service failures and removes failed instances from the load balancing pool, redirecting traffic to healthy instances. Health check intervals and removal thresholds can be configured based on operational requirements.

**Database Performance Through Proper Indexing**: The MySQL database implementation includes indexes on frequently queried columns (user_id, order_id, payment_id) enabling efficient retrieval of orders by user or payment by order without scanning entire tables. Index selection was informed by query patterns—columns appearing frequently in WHERE clauses and JOIN conditions are indexed. Over-indexing is avoided as indexes consume storage and slow down write operations, creating a performance trade-off.

**Frontend Performance Through Reactive Rendering**: React's virtual DOM and reconciliation algorithm ensure that only changed portions of the UI are redrawn, avoiding expensive full-page redraws. TypeScript provides compile-time type checking preventing runtime errors. Code splitting enables loading only the JavaScript required for the current page, reducing initial load times. Tailwind CSS provides utility-first styling enabling rapid development without context-switching to CSS files.

**Security Layers Through Defense-in-Depth**: Multiple security layers protect against different attack vectors. Application-layer security (Spring Security, JWT validation) protects against unauthorized access. Database-layer constraints (unique constraints, check constraints) prevent invalid data. Transport-layer security (HTTPS) prevents network eavesdropping. Input validation prevents injection attacks. Role-based access control restricts sensitive operations. This layered approach ensures that compromise of one layer does not expose the entire system.
